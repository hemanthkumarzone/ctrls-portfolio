Infrastructure security

Our platform runs on AWS with multi-region redundancy. All infrastructure is provisioned and managed through infrastructure-as-code with version-controlled configurations, mandatory change review, and automated compliance checks on every deployment.

🌍

Multi-region architecture

Production workloads are distributed across AWS US-East, EU-West, and AP-Southeast regions. Data residency is enforced at the infrastructure level — your data never leaves your chosen region.

⚙️

Infrastructure as code

All infrastructure is defined in Terraform and deployed through CI/CD pipelines with mandatory peer review, security scanning, and automated policy checks before any change reaches production.

🔄

Immutable deployments

We use immutable infrastructure patterns — servers are never patched in place. Every deployment replaces instances entirely, eliminating configuration drift and reducing attack surface.

💾

Backup and recovery

All customer data is backed up continuously to geographically separate AWS regions. Backups are encrypted, integrity-checked hourly, and tested monthly. RTO is under 4 hours; RPO is under 1 hour.

🔒

Data security and encryption

Every piece of data you entrust to the platform — billing telemetry, workload metadata, cost attribution records, orchestration logs — is encrypted at every stage of its lifecycle.

Data stateEncryption standardKey management
Data in transit TLS 1.3 (minimum TLS 1.2). TLS 1.0/1.1 disabled. Managed certificates via AWS ACM with auto-renewal
Data at rest (databases) AES-256 encryption on all RDS and DynamoDB instances AWS KMS managed keys (default) or BYOK (Enterprise)
Data at rest (object storage) AES-256 encryption on all S3 buckets. Bucket policies enforce encryption. AWS KMS with per-customer key isolation on Enterprise tier
Data at rest (backups) AES-256 with separate encryption keys from production AWS KMS with cross-region key replication
Agent telemetry in transit TLS 1.3 with certificate pinning. Agent refuses connections to unverified endpoints. Short-lived mTLS client certificates rotated every 24 hours
Secrets and credentials Never stored in code or environment variables. Stored in AWS Secrets Manager. Automatic rotation every 30 days for all platform secrets

Bring your own key (BYOK)

Enterprise customers can supply their own AWS KMS keys (or keys from Azure Key Vault and HashiCorp Vault) for all data encryption. BYOK means only you can decrypt your data — even if our systems were compromised, your data remains inaccessible without your key. Revoking your key immediately renders all stored data unreadable.

We maintain strict data isolation between customers at both the application and infrastructure layer. Your data is logically and physically separated from all other customers — we do not use shared databases or storage buckets across customer accounts.

Access control and identity

Access to customer data within our systems is controlled by a strict least-privilege model. No engineer has standing access to production customer data — all access is time-limited, requires justification, and is fully logged.

🔑

Zero standing access

No AIFinOps employee has permanent access to production systems or customer data. Access is granted on-demand via a just-in-time (JIT) access system, requires manager approval, and expires automatically.

📋

Immutable audit logs

Every access event — including all read operations on customer data — is written to tamper-proof, append-only audit logs in AWS CloudTrail. Logs are retained for 3 years and available for your review on request.

📱

MFA enforced internally

MFA using hardware security keys (YubiKey) is mandatory for all AIFinOps employees accessing production systems. TOTP-based MFA is also available but not accepted for high-privilege operations.

👤

Role-based access control

Internal roles follow a strict least-privilege model reviewed quarterly. Engineers cannot deploy to production without a code review, and deployment pipelines enforce separation of duties — the author cannot approve their own change.

Customer-facing access controls

  • RBAC with custom roles — define exactly what each team member can view, configure, or automate within your account.
  • SSO via SAML 2.0 and OIDC — integrate with Okta, Azure AD, Google Workspace, and other identity providers.
  • MFA support — TOTP and hardware security keys supported for all account users. MFA can be enforced organisation-wide from your admin settings.
  • Session management — configurable session timeouts, concurrent session limits, and forced re-authentication for sensitive operations.
  • API key scoping — API keys can be scoped to read-only, specific capabilities, or specific connected environments. Keys are rotatable and revocable at any time.
  • IP allowlisting — restrict Platform access to specific IP ranges for additional defence-in-depth.

Network security

Our network architecture is designed to minimise attack surface, isolate components, and detect threats in real time at every layer.

🛡️

DDoS protection

All public endpoints are protected by AWS Shield Advanced with automatic detection and mitigation of volumetric, protocol, and application-layer DDoS attacks.

🔥

Web application firewall

AWS WAF is deployed in front of all public APIs and the web application, with managed rule sets for OWASP Top 10, bot detection, and custom rate limiting rules.

🏠

VPC isolation

All production services run in isolated VPCs with no direct public internet access. Traffic between services travels over private endpoints. Internet-facing services are restricted to load balancers and API gateways only.

🔎

Intrusion detection

AWS GuardDuty monitors all VPC flow logs, DNS queries, and CloudTrail events for anomalous activity. Alerts trigger automated containment and human review within 15 minutes during business hours, 30 minutes at all other times.

Agent security

The AIFinOps Agent is a lightweight binary you install in your on-premises or private cloud environment to collect hardware and workload telemetry. Its security model is designed around minimal footprint and zero trust.

📦

Minimal permissions

The Agent runs as a non-root, dedicated system user with read-only access to metrics APIs (DCGM, Prometheus, /proc). It cannot write to your filesystem, execute commands, or modify any configuration.

📡

Outbound-only communication

The Agent initiates outbound-only connections over TLS 1.3 to a fixed set of AIFinOps endpoints (published at docs.aifinops.com/agent/endpoints). It accepts no inbound connections and opens no ports.

🔏

Code signing

Every Agent release is signed with our GPG key. The installer verifies the signature before executing. Signatures and checksums are published at aifinops.com/agent/verify for independent verification.

🔄

Automatic updates with approval

Agent updates are delivered automatically by default but can be set to manual approval mode in enterprise environments. All updates maintain backward compatibility with a 6-month deprecation notice for breaking changes.

🚫

No data residency outside your network

The Agent holds a maximum 60-second metric buffer in memory only — nothing is written to disk. If the outbound connection is unavailable, buffered data is dropped, not queued to disk.

🧪

Open for audit

The Agent's source code is available for review under NDA for enterprise customers. We also publish a detailed technical specification of every system call and network request the Agent makes.

You can review the exact permissions the Agent requires, inspect its network traffic, and uninstall it at any time without data loss — the Agent holds no persistent state. Full technical documentation at docs.aifinops.com/agent/security.

Vendor and supply chain security

Our platform is only as secure as the vendors we depend on. We apply a rigorous vendor risk management process to every third party that has access to customer data or our production systems.

  • All sub-processors are reviewed annually against a 50-point security checklist covering data handling, encryption, access controls, incident response, and compliance certifications.
  • Third-party software dependencies are scanned continuously using Dependabot and Snyk. Critical and high vulnerabilities are patched within 24 hours; medium vulnerabilities within 7 days.
  • Our software build pipeline is isolated from the internet, uses reproducible builds, and generates an SBOM (software bill of materials) for every release — available to enterprise customers on request.
  • All production code changes require two independent code reviews, automated SAST (static application security testing), and secrets scanning before merge. No single engineer can deploy to production alone.
  • A full and current list of sub-processors is published at aifinops.com/legal/sub-processors. Customers are notified 14 days before any new sub-processor with access to customer data is added.

Incident response

We maintain a documented and tested incident response plan. In the event of a security incident affecting your data, we will notify you promptly and transparently — before we have all the answers, not after.

0 – 1 hr

Detection and triage

Automated monitoring detects the incident. On-call security engineer is paged. Incident is classified by severity (P1–P4) and an incident commander is assigned for P1/P2 events.

1 – 4 hrs

Containment

Affected systems are isolated. Scope of impact is assessed. If customer data may be involved, the legal and privacy teams are engaged. Initial internal communication is sent.

Within 24 hrs

Customer notification (if impacted)

Affected customers receive an initial notification via email and in-platform alert. We share what we know, what we don't yet know, and what steps we are taking — even if the investigation is ongoing.

Within 72 hrs

Regulatory notification

Where a personal data breach is confirmed, we notify the relevant supervisory authority (e.g. Ireland DPC for GDPR) within 72 hours as required by GDPR Article 33.

Post-incident

Root cause analysis and remediation

A blameless post-mortem is conducted within 5 business days. A written root cause analysis (RCA) report is shared with affected customers. Remediation actions are tracked to completion.

Our incident response plan is tested through tabletop exercises twice a year and a live red team exercise annually. Test results inform improvements to our detection, containment, and communication procedures.

Responsible disclosure

We believe security researchers play a vital role in keeping platforms like ours safe. If you discover a vulnerability in our platform, we ask that you report it to us responsibly so we can fix it before it is exploited.

Our commitment to researchers

  • We will acknowledge your report within 1 business day.
  • We will provide an initial assessment of severity and timeline within 5 business days.
  • We will not pursue legal action against researchers who discover and report vulnerabilities in good faith and in accordance with this policy.
  • We will credit researchers in our security acknowledgements page (with permission) when their finding results in a confirmed fix.
  • Rewards are available through our private bug bounty programme for qualifying vulnerabilities. Contact security@aifinops.com to request an invitation.

Scope

  • In scope: aifinops.com web application, API (api.aifinops.com), the AIFinOps Agent binary, and the aifinops.com marketing site.
  • Out of scope: third-party services we use (AWS, Stripe, Intercom), social engineering attacks targeting our employees, physical security testing, and denial-of-service attacks.

Report vulnerabilities to security@aifinops.com. Please encrypt your report using our PGP key, available at aifinops.com/security/pgp.asc. Do not disclose the vulnerability publicly until we have confirmed a fix is in place.
