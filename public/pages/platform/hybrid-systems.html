Frequently asked questions
What costs does the TCO module capture that standard cloud billing tools miss? +
Standard cloud billing tools only track OpEx — what you pay the cloud provider each month. The TCO module additionally models on-premise CapEx amortised over the hardware lifecycle, power consumption using real PUE coefficients, cooling and facilities allocation, network infrastructure depreciation, data centre staff costs, maintenance contracts, and end-of-life disposal. These hidden costs typically add 40–50% to the apparent on-prem hardware price and are invisible without a dedicated TCO model.
How does the platform calculate the breakeven point between on-prem and cloud GPU? +
The breakeven calculator compares the fully-loaded monthly on-premise TCO per GPU — including depreciation, power, cooling, and operations — against the equivalent cloud GPU cost at the same utilisation rate. It applies configurable depreciation curves (straight-line, declining balance, or custom) and adjusts for actual cluster utilisation. The output is a month-by-month crossover chart showing exactly when on-prem becomes cheaper, with sensitivity analysis for utilisation rate changes.
Can the module support multi-site and international datacenter environments? +
Yes. The module supports multiple datacenter sites with site-specific PUE values, local electricity tariffs in any currency, regional carbon intensity factors, and country-specific staffing cost models. Costs are normalised to a common currency for cross-site comparison and can be segmented by site, region, or business unit in all reports and dashboards.
How does workload placement optimisation work in practice? +
The placement engine continuously evaluates each active AI workload against a cost model for every available infrastructure tier. For each workload it considers: duration and recurrence pattern, GPU memory and compute requirements, data residency constraints, SLA priority class, and real-time cost per GPU-hour on each tier. It then surfaces a recommendation — move this training job to on-prem idle capacity, run this inference workload on cloud spot — with the projected monthly saving for each action.
How is power usage effectiveness (PUE) factored into the cost model? +
PUE is applied as a multiplier on raw server power draw to account for overhead energy consumed by cooling, lighting, and power distribution. The module accepts actual measured PUE values per site or uses industry benchmarks (1.2 for modern hyperscale, 1.5–1.8 for legacy facilities). Power cost is then calculated as: (server watt-hours × PUE × kWh rate) and attributed to each GPU rack and workload proportionally.
Does the module support ESG and carbon reporting alongside financial TCO? +
Yes. The module appends a carbon dimension to every TCO calculation by multiplying energy consumption (kWh) by the regional grid carbon intensity (gCO2/kWh) sourced from live emissions databases. Reports include total carbon cost in tonnes CO2e per infrastructure layer, allowing organisations to compare the carbon footprint of on-prem versus cloud workloads and satisfy ESG disclosure requirements including GHG Protocol Scope 2 and 3 reporting.
Can finance teams use the module without deep Kubernetes or cloud knowledge? +
This is a core design goal of the module. Finance users see costs expressed in familiar terms — monthly spend, depreciation schedules, cost centre allocations, and variance reports — without needing to understand Kubernetes namespaces or cloud SKU naming conventions. A separate engineering view provides the technical layer. Both views pull from the same underlying data model so reports are always consistent across teams.
How does the platform handle hardware refresh cycles and procurement planning? +
The module tracks the age and depreciation status of every on-prem hardware asset. When a GPU generation approaches end of productive life — typically year 3–4 for AI accelerators — the platform generates a procurement scenario comparing: full refresh with new hardware, extending the current asset with cloud burst capacity, migrating fully to cloud, or a hybrid lease arrangement. Each scenario includes a 3-year and 5-year TCO projection to support capital budget planning cycles.
