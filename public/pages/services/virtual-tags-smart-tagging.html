The problem

Untagged resources are
unattributed spend.

Cloud providers and AI platforms generate thousands of billable line items per month. Your cost attribution is only as good as your tagging coverage — and in practice, tagging is an afterthought. Pods get deployed without labels, API keys get shared across teams, and billing line items carry no project context. The result: a growing pile of "unattributed" spend that no one owns and no one fixes.

The conventional solution is to mandate better tagging and train engineers to apply labels consistently. This approach has a 100% failure rate at scale. Engineers have ship deadlines. Tags get forgotten. And by the time the unattributed spend is discovered in the monthly report, it's weeks old.

Virtual Tags and Smart Tagging take the opposite approach: instead of asking humans to tag perfectly, we infer tags automatically from every available signal — naming conventions, deployment metadata, git repository ownership, IAM roles, team membership, and usage patterns — and apply them in the platform layer, without touching a single production resource.

Virtual tags exist only in the AIFinOps platform. They never modify your cloud resources, Kubernetes manifests, or infrastructure-as-code. Zero risk, immediate attribution improvement.

Virtual Tags — tags that exist only in the platform

Virtual Tags are attribution dimensions that you define inside the AIFinOps platform and apply to cost items — without touching your cloud resources, Kubernetes manifests, or infrastructure-as-code. They let you build an attribution model that reflects your organisation's actual structure, not just what your cloud provider supports.

A cloud bill doesn't know about your "Payments" product line, your "AI Platform" cost centre, or whether a workload is "customer-facing" vs "internal." Virtual Tags let you define these dimensions and apply them to your cost data — so your reports reflect your business, not your cloud topology.

Create any tag dimension your org needs — team, product, use-case, customer segment, compliance scope, strategic initiative
Apply virtual tags using rule-based matching — by existing tag values, resource names, account IDs, namespace patterns, or cost thresholds
Retroactively apply virtual tags to historical cost data — rewrite your entire attribution history without touching production
Override incorrect or inconsistent existing tags without modifying your infrastructure — platform-level corrections that don't require a deployment
Every virtual tag is versioned — changes are tracked, and you can see how your attribution model evolved over time

Virtual tag rule types

Six rule types for any
attribution scenario

Every virtual tag is defined by a rule that maps conditions to tag values. Six rule types cover every attribution scenario — from simple pattern matching to complex multi-condition logic.

Pattern match

Resource name pattern

Apply a virtual tag when a resource name matches a regex or wildcard pattern — the most common rule type for organisations with consistent naming conventions.

# tag all pods with "search" in the name
if resource.name matches "*search*":
  apply product = "search-ai"
  apply team = "search-eng"
Inheritance

Tag inheritance

Propagate one or more tags from a parent resource (namespace, account, subscription) to all child resources — filling the most common gap automatically.

# inherit from namespace to all pods
if namespace.tag["team"] exists:
  inherit team → all child pods
  inherit cost-centre → all child pods
Lookup table

Lookup table mapping

Map a specific tag value or resource attribute to a set of virtual tags using an uploaded CSV or inline mapping table — ideal for batch-correcting known mis-tagged resources.

# map existing team names to cost centres
lookup tag["team"] in csv:
  search-eng → cc: "cc-search"
  ml-platform → cc: "cc-infra"
Condition

Multi-condition rule

Apply virtual tags only when multiple conditions are all true — AND/OR logic across any combination of resource attributes, existing tags, cost thresholds, or time patterns.

# tag large training jobs differently
if resource.type == "Job"
  AND resource.gpu_count > 4:
  apply workload-tier = "large-training"
Tiered

Cost-threshold tiering

Apply different virtual tag values based on a resource's monthly cost — automatically segmenting workloads into cost tiers for differentiated governance and reporting.

# tier workloads by monthly cost
if monthly_cost > 5000:
  apply cost-tier = "high"
elif monthly_cost > 1000:
  apply cost-tier = "medium"
Composite

Composite tag assembly

Build new virtual tag values by combining parts of existing tags or attributes — for example, assembling a unique workload identifier from team + product + environment.

# build a composite workload ID
compose workload-id =
  tag["team"] + "-"
  + tag["product"] + "-"
  + tag["env"]

All virtual tag rules are version-controlled and retroactive — a new rule applies to your entire cost history immediately, not just future data. Every historical report updates automatically.

Tag governance

Tags only stay accurate
if they're governed

Creating tags is the first step. Keeping them accurate as your infrastructure evolves requires ongoing governance — automated enforcement, drift detection, and policy tools.

📋

Required tag policies

Define which tags are mandatory for all resources, per namespace, or per resource type. New resources missing required tags trigger immediate alerts — before cost accumulates unattributed.

📉

Coverage drift alerts

Automated alerts when tag coverage drops below a configurable threshold — catching new untagged deployments within 15 minutes rather than discovering them in the next monthly report.

🔄

Tag standardisation enforcement

Define allowed values for each tag key (e.g. environment must be "prod", "staging", or "dev" — not "production", "Production", or "prd") and auto-normalise non-standard values via virtual tag correction rules.

📊

Tag coverage reporting

Per-namespace, per-team, and per-tag-key coverage reports showing attribution completeness, trend over time, and which resources are responsible for coverage gaps.

🔗

Inheritance chain visualiser

See exactly where each virtual tag value came from — which rule applied it, which parent it was inherited from, and what confidence score the inference carried.

⚡

Terraform & IaC integration

Optional: integrate with your Terraform or Helm workflows to validate that required tags are present before deployment — preventing gaps from entering production in the first place.
