Detection engine

ML-powered baselines that learn
your spending patterns

Static threshold alerts — "alert when spend exceeds $X" — produce two failure modes: they miss real anomalies that stay under the threshold, and they fire false positives every time a workload legitimately scales. Both destroy trust.

Our detection engine builds a dynamic behavioural model for every monitored signal using a combination of statistical and machine learning techniques. Baselines adapt to weekly cycles, seasonal patterns, and growth trends — so an anomaly is always defined relative to what is normal for that specific workload at that specific time.

The result is 99.2% detection accuracy with a false positive rate under 0.4% — meaning your team responds to alerts that matter, not noise.

Baselines are built per workload, not across customers. Your spending patterns are never compared to or influenced by other organisations' data.

Core
STL decomposition

Seasonal-Trend decomposition using LOESS — separates trend, seasonal, and residual components to define what is anomalous versus cyclically expected for any signal.

Core
Isolation Forest

Unsupervised anomaly detection for multi-dimensional signals — detects anomalies in GPU spend, request rate, and cost simultaneously without requiring labelled training data.

Predictive
Prophet forecasting

Time-series forecasting to detect when the current trajectory will breach a threshold before it happens — enabling proactive intervention, not reactive firefighting.

Streaming
CUSUM change detection

Cumulative Sum algorithm for detecting gradual cost drift that traditional threshold alerts miss — catches slow-moving anomalies before they become significant overruns.

Correlation
Cross-signal correlation

Combines cost, usage, performance, and infrastructure signals to distinguish legitimate scale events from true anomalies — reducing false positives by 85% versus single-signal detection.

The baseline model uses STL decomposition to understand seasonal patterns (including event-driven spikes like product launches) alongside cross-signal correlation. A cost spike accompanied by a proportional increase in request volume and active user traffic is classified as legitimate scale — not an anomaly. An identical cost spike with flat or declining request volume is flagged immediately. You can also define "planned event" windows where anomaly thresholds are temporarily relaxed, and the model learns from these windows to adjust future baselines.
Circuit breakers only execute within the permission scope you configure during setup. By default, all circuit breakers are notification-only — they alert your team but take no automated action. You explicitly enable each action type (rate-limit, throttle, pause, cap) per workload or per team, and set the specific parameters. A circuit breaker can never expand beyond the action types you've authorised. Every action is logged with the exact trigger condition and can be reviewed and reversed at any time.
Statistical baselines are available within 24 hours of connecting a workload — sufficient for high-confidence spike detection. Full seasonal decomposition models (which handle weekly and monthly cycles) require 14 days of data. For new workloads without history, we apply a conservative initial threshold using cross-workload pattern matching from similar workload types in your environment — this provides meaningful anomaly detection from day one while the personalised baseline develops.
