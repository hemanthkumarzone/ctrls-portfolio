Why it matters

Total spend is a symptom.
Unit cost is the diagnosis.

Every engineering decision that affects AI cost — model selection, context window length, caching strategy, batching, GPU provisioning — shows up in your unit cost. But those signals are invisible in a monthly cloud bill. A bill that goes from $80K to $100K tells you something changed. Unit cost tells you what changed and gives you the lever to fix it.

Unit Economics transforms AI cost from a CFO problem into a product problem — and product teams know how to optimise. When cost-per-inference is a metric on the product dashboard alongside latency and error rate, engineers naturally start asking "how do I reduce it?" the same way they ask "how do I reduce P99?"

The most financially sustainable AI products — those that scale without proportionally scaling cost — are built by teams that track and optimise unit economics from the earliest stage. Our platform makes that discipline available from day one, with zero instrumentation required.

A unit cost reduction of 30% on a service processing 50M inferences per month translates directly to the bottom line — regardless of what the total monthly bill says.

Compute component 44% · $0.00185
CPU, memory, and node pool costs allocated to this service via namespace attribution. This is the component most directly affected by right-sizing — reducing CPU requests from 8 to 3 vCPU cuts this component by ~60%.
Model API component 32% · $0.00134
Token costs from the upstream model API — input tokens × input price + output tokens × output price, net of cache savings. The lever here is prompt engineering (shorter inputs), output length control, and prompt caching strategy.
GPU amortisation 16% · $0.00067
For self-hosted or GPU-accelerated inference, the amortised GPU hardware cost per inference — calculated from total GPU cost divided by total inference volume on those GPUs. Improving GPU utilisation reduces this directly.
Shared overhead 8% · $0.00034
Allocated share of cluster overhead (monitoring, ingress, DNS, logging) using the proportional allocation policy. This component moves slowly and is the least actionable — but it should be visible in every unit cost for full transparency.

The waterfall makes optimisation prioritisation obvious: a 30% reduction in the compute component saves more per inference than a 30% reduction in the overhead component — because compute is 44% of the unit cost vs 8%.

🔀 Orchestration The orchestration engine uses unit cost signals to make routing decisions — scheduling batch workloads during periods when spot instance pricing drives down the compute component of unit cost, and routing to the lowest-cost-per-inference model endpoint that meets the latency SLA. Routing is optimised for unit cost, not just total cost. Orchestration actively reduces unit cost by routing to cost-efficient compute rather than just any available compute. Cost-per-inference improvements from orchestration are tracked separately and attributed to the orchestration engine for FinOps programme reporting.
🗑️ Waste elimination Idle resources inflate unit cost by adding fixed cost to the denominator without increasing the inference volume denominator. A pod consuming 30% of its allocated resources is inflating the compute component of unit cost by approximately 3×. Waste elimination is unit cost reduction — surfaced in unit cost terms so the impact is immediately visible. Every waste elimination action shows its unit cost impact: "Retiring this idle pod will reduce cost-per-inference from $0.0042 to $0.0038." Teams can see the unit economics impact of waste before approving any elimination action.

Common questions

Unit cost calculation is a multi-source attribution process. Compute cost comes from Kubernetes cluster billing and is attributed to the specific service via namespace-level cost allocation. Model API token cost comes from the provider's billing API (OpenAI, Anthropic, etc.) and is attributed using API key ownership mapping. GPU cost comes from either cloud billing or DCGM utilisation data for reserved hardware. Shared overhead is allocated using your configured allocation policy. All components are normalised to a per-inference basis by dividing by the actual inference count measured from the service's request metrics. The calculation runs every 15 minutes and is fully auditable — you can see exactly how every component was derived.
Yes. Unit cost history is available from the moment you connect your data sources. For all cost components that existed before your connection date, retroactive calculation is available using your historical billing data — most customers can view 12–24 months of unit cost history immediately after connecting. Unit cost trend charts show the trajectory of each metric alongside the specific platform actions (right-sizing applied, model routing changes, waste eliminated) that caused changes — so you can attribute unit cost improvements to specific decisions.
For workloads where "inference" is not a clean unit, we support alternative unit definitions: for batch jobs, the unit can be "records processed" (cost per 1,000 records processed), "job run" (cost per job execution), or "epoch" (cost per training epoch). For streaming workloads, the unit can be "event processed" or "context update." You configure the unit definition per workload type, and the platform measures that unit's volume from the appropriate source (job completion events, Kafka consumer lag metrics, etc.) and divides total cost accordingly. Any countable business event that your infrastructure can emit can become a unit cost denominator.
