Why it's hard

Kubernetes was designed for scheduling,
not cost accounting

In a virtual machine world, cost attribution is simple: one VM, one team, one bill. Kubernetes destroys that model. Pods are ephemeral, nodes are shared, GPUs are time-sliced, storage is dynamic, and namespaces are the only organisational boundary — but they were designed for access control, not financial ownership.

Standard cloud billing tools see a node. They cannot see which pods ran on it, which team owns those pods, how much of the node's GPU was used by which workload, or how much of the node cost should be allocated as shared overhead versus directly attributed to a specific team.

Our attribution engine solves all of this. It operates at the metrics API level — reading real resource consumption per pod, per container, per GPU process — and maps that to real billing data from your cloud provider to produce accurate, auditable cost attribution down to the individual pod and team.

Attribution accuracy matters: organisations that cannot attribute Kubernetes costs cannot hold teams accountable, cannot forecast accurately, and cannot identify waste. Estimation-based tools produce numbers no finance team will trust.

01
Request-based allocation

Allocates node cost by the ratio of a pod's resource requests to total node capacity. Fast to compute, but systematically overcharges teams that over-provision and undercharges those that right-size.

Imprecise
02
Usage-based allocation

Allocates based on actual CPU, memory, and GPU consumed per pod per time interval. More accurate, requires continuous telemetry collection from cAdvisor and DCGM.

Recommended
03
Blended allocation

Combines usage-based attribution for compute-intensive workloads with request-based for storage and network — producing the most accurate multi-resource cost breakdown.

Our default
04
Shared cost allocation

Handles cluster-level overhead — ingress, DNS, monitoring, logging — and distributes it proportionally across tenants using configurable sharing policies.

Configurable

Allocation policies

Four ways to split shared costs.
You choose.

Shared cluster infrastructure — ingress controllers, monitoring stacks, DNS, logging — needs to be allocated fairly. We support four policies, configurable per component.

Proportional

Resource-proportional

Distribute shared cost proportionally by each tenant's share of total resource consumption (CPU + memory + GPU). Most accurate for heterogeneous workloads.

shared_cost × (team_resource / total_resource)
Equal split

Even distribution

Split shared costs equally across all active namespaces. Simplest model for teams at similar scale — avoids penalising heavy users of cluster-level services.

shared_cost / active_namespace_count
Weighted

Custom weight allocation

Assign explicit percentage weights to each namespace or team for shared cost distribution — useful when business agreements or SLAs define cost responsibility.

shared_cost × team_weight_pct
Direct

Direct overhead exclusion

Exclude shared costs from team attribution entirely and report them as a separate platform overhead line — common for internal platform teams who don't chargeback shared infra.

platform_overhead_pool (not allocated)
